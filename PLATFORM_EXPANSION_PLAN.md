# 플랫폼 확장 계획안

## 1. 문서 목적

이 문서는 현재 `swimming-diary-dashboard` 프로젝트를 기반으로,

- 수영 중심의 기록 대시보드를
- 동호회용 운동 플랫폼으로 확장하고
- 모바일, 앱, PC 어디서든 쉽게 쓸 수 있게 만들며
- 수영, 러닝, 싸이클까지 확장 가능한 모듈형 구조로 재설계하기 위한 방향을 정리한 계획안이다.

핵심 목표는 "누구나 쉽게 구성할 수 있는 동호회용 운동 플랫폼"이다.

---

## 2. 현재 프로젝트 상태 요약

현재 프로젝트는 단순한 화면 시안 수준이 아니라, 이미 아래 요소를 갖춘 운영형 대시보드에 가깝다.

### 2-1. 현재 구현된 핵심 기능

- DCInside `swimmingdiary` 게시판 크롤링
- 게시글 제목 기반 운동 기록 파싱
- 기록 집계 및 누적 통계 생성
- 랭킹 생성
- 개인 프로필 페이지 생성
- 배지/칭호 계산
- 파싱 실패 검토 큐
- 관리자 페이지 및 설정 저장 API
- JSON 기반 운영 설정 구조
- Docker/NAS 배포 자산

### 2-2. 현재 데이터 파이프라인 구조

현재 구조는 이미 플랫폼 확장에 유리한 4단계 흐름을 가지고 있다.

1. `crawler.py`
   외부 소스에서 글/기록을 수집
2. `parser.py`
   원문을 규칙에 맞게 기록 데이터로 변환
3. `aggregate.py`
   기록을 집계해서 요약, 랭킹, 프로필, 배지 데이터를 생성
4. `pipeline.py`
   공개용 JSON 파일과 운영용 산출물을 생성

즉, 지금 프로젝트는 "수영 게시판 전용 대시보드"이지만 내부 구조는 이미 "수집 엔진 + 규칙 엔진 + 집계 엔진 + UI" 형태로 나뉘어 있다.

### 2-3. 현재 공개 산출물

현재 프로젝트는 이미 화면에서 바로 사용할 수 있는 공개 데이터 파일을 생성한다.

- `docs/data/summary.json`
- `docs/data/leaderboard.json`
- `docs/data/dashboard_views.json`
- `docs/data/author_index.json`
- `docs/data/author_profiles.json`
- `docs/data/badge_index.json`
- `docs/data/parse_status.json`
- `docs/data/review_queue.json`

이 구조는 향후 REST API 또는 앱 백엔드 응답 구조의 초안으로 그대로 발전시킬 수 있다.

### 2-4. 현재 운영/설정 구조

현재 프로젝트는 운영자가 직접 바꿀 수 있는 설정 파일 구조도 이미 갖고 있다.

- `data/admin/site_config.json`
- `data/admin/navigation_config.json`
- `data/admin/home_sections.json`
- `data/admin/badge_catalog.json`
- `data/admin/season_badges.json`
- `data/admin/gallery_title_rules.json`
- `data/admin/profile_layout_config.json`
- `data/admin/badge_art_catalog.json`

이 부분은 향후 "플랫폼 빌더" 개념의 기초가 된다.
즉, 코드를 수정하지 않고도 동호회별 화면, 메뉴, 배지, 운영 규칙을 바꾸는 방향으로 이미 출발해 있다.

### 2-5. 현재 확인된 운영 지표

`docs/data/*.json` 기준 현재 프로젝트는 아래 정도의 실제 운영 데이터를 다루고 있다.

- 집계 기준일: `2026-03-17`
- 전체 보이는 글 수: `122`
- 자동 반영된 운동 기록 수: `44`
- 활성 작성자 수: `13`
- 누적 거리: `68,825m`
- 파싱 대기/검토 큐: `78`
- 현재 자동 반영 성공률: `36.07%`

즉, 지금 단계는 "기능 검증용 프로토타입"을 넘어 "운영 흐름 검증이 가능한 초기 제품" 단계로 볼 수 있다.

## 3. 현재 프로젝트의 한계

플랫폼으로 확장하려면 현재 구조의 한계도 분명히 봐야 한다.

### 3-1. 데이터 입력 방식 한계

현재는 게시글 제목 형식 `거리 / 시간` 에 매우 강하게 의존한다.

- 사용자 입력 부담이 큼
- 다른 운동 앱과 연결되지 않음
- 자동화 확장성이 낮음
- 수영 외 종목 확장 시 유지비가 커짐

### 3-2. 단일 커뮤니티/단일 종목 구조

현재는 사실상 아래 조건에 최적화되어 있다.

- 단일 커뮤니티
- 수영 전용 규칙
- 단일 데이터 소스
- 단일 시즌/단일 집계 방식

플랫폼이 되려면 여기서 다음 구조로 바뀌어야 한다.

- 멀티 클럽
- 멀티 스포츠
- 멀티 입력 채널
- 클럽별 규칙 커스터마이징

### 3-3. 운영 방식 한계

현재는 정적 JSON 기반 대시보드와 관리자 저장 중심이다.

플랫폼 단계에서는 다음이 추가되어야 한다.

- 사용자 계정
- 클럽 생성/가입
- 권한 체계
- 모임/이벤트 관리
- 신청/참가 관리
- API 기반 실시간 또는 준실시간 동기화

## 4. 플랫폼 비전

### 한 줄 정의

"운동 기록, 랭킹, 배지, 목표, 모임 운영을 하나로 묶은 동호회용 모듈형 스포츠 플랫폼"

### 플랫폼이 제공해야 하는 가치

- 회원은 자기 운동 기록과 성장 흐름을 쉽게 본다.
- 운영진은 클럽 대시보드와 모임 관리를 쉽게 한다.
- 클럽은 배지, 챌린지, 목표 규칙을 자기 스타일에 맞게 조립한다.
- 종목은 수영에서 시작해 러닝, 싸이클로 쉽게 확장한다.

### 핵심 사용자

- 운동 동호회 운영자
- 동호회 일반 회원
- 소규모 크루/스터디 리더
- 기록 중심 커뮤니티를 만들고 싶은 개인 운영자

---

## 5. 목표 제품 구조

향후 제품은 아래 6개 모듈로 설계하는 것이 적합하다.

### 5-1. 기록 수집 모듈

역할:

- 운동 결과 앱/사이트/API 연동
- 수동 기록 입력
- 파일 업로드 또는 스크린샷 보조 입력

초기 연결 대상 예시:

- 수영 기록 앱
- 러닝 앱
- 싸이클 앱
- Apple Health / Google Fit 연동 가능성 검토

핵심 방향:

- "파싱" 중심에서 "API 수집 + 표준화" 중심으로 전환

### 5-2. 종목 표준화 모듈

역할:

- 수영, 러닝, 싸이클 기록을 공통 포맷으로 변환

예시 공통 필드:

- `sport_type`
- `activity_date`
- `distance_m`
- `duration_sec`
- `pace`
- `avg_hr`
- `calories`
- `source_app`
- `source_activity_id`

추가 종목별 확장 필드:

- 수영: 영법, 풀 길이, 랩 수
- 러닝: 고도, 케이던스
- 싸이클: 파워, 고도, 평균 속도

### 5-3. 분석/집계 모듈

역할:

- 개인 기록 분석
- 클럽 누적 통계
- 랭킹
- 성장 추이
- 시즌별 비교

현재 프로젝트의 `aggregate.py` 는 이 모듈의 훌륭한 출발점이다.

### 5-4. 재미 요소 모듈

역할:

- 배지
- 칭호
- 챌린지
- 목표 달성
- 시즌 이벤트
- 개인 역량 리포트

현재 프로젝트의 배지 시스템은 이 모듈의 1차 버전으로 활용 가능하다.

### 5-5. 클럽 운영 모듈

역할:

- 모임 일정 생성
- 참가 신청
- 참석 체크
- 공지
- 시즌/챌린지 운영

이 모듈은 현재 프로젝트에 없는 영역이므로, 플랫폼화 과정에서 새로 추가해야 한다.

### 5-6. 플랫폼 빌더 모듈

역할:

- 클럽별 메뉴 구성
- 홈 대시보드 조립
- 종목 선택
- 배지 규칙 선택
- 목표 방식 선택
- 모임 운영 방식 설정

현재 `data/admin/*.json` 구조는 이 빌더 모듈의 초기 설계 자산으로 재사용 가능하다.

## 6. 권장 플랫폼 아키텍처

현재 구조를 유지하면서 확장하려면 아래 방향이 현실적이다.

### 6-1. 1단계 리팩터링 방향

기존:

- `DcinsideCrawler`
- 제목 파서
- 수영 전용 집계

확장 후:

- `connectors/`
  외부 서비스별 수집기
- `normalizers/`
  종목별 표준화 로직
- `engines/aggregation`
  집계 엔진
- `engines/badges`
  배지/칭호 엔진
- `engines/challenges`
  목표/챌린지 엔진
- `club_builder/`
  클럽 설정/화면 구성

### 6-2. 권장 도메인 분리

- `users`
  사용자/프로필/권한
- `clubs`
  클럽/멤버십/역할
- `sports`
  종목 정의
- `activities`
  운동 기록 원본/정규화 데이터
- `achievements`
  배지/칭호/목표
- `events`
  모임/훈련/참가신청
- `leaderboards`
  랭킹/비교/시즌 보드

### 6-3. 데이터 저장 방향

현재 JSON 산출물은 유지하되, 플랫폼 단계에서는 DB 중심 구조로 가야 한다.

권장:

- 운영 DB: PostgreSQL
- 캐시/큐: Redis
- 공개 API 응답 캐시: JSON 또는 CDN

초기에는 아래 방식이 무난하다.

- 원본 활동 데이터: DB
- 집계 결과: DB + 공개용 JSON 캐시

## 8. 단계별 실행 로드맵

## Phase 0. 현재 프로젝트를 플랫폼 베이스로 정리

목표:

- 현재 수영 대시보드를 "플랫폼 코어" 관점으로 재구성

할 일:

- 수영 전용 명칭을 공통 용어로 치환
- 크롤러/파서/집계/배지/운영 설정 폴더를 도메인별로 재배치
- 공개 JSON 스키마를 API 응답 스키마 관점으로 정리
- `record` 모델을 `activity` 중심으로 일반화

결과:

- 기존 프로젝트가 확장 가능한 코어 엔진이 됨

## Phase 1. 입력 방식 전환

목표:

- 게시글 파싱 중심에서 API/수동입력 중심으로 전환

할 일:

- `source_connector` 인터페이스 설계
- 수영 1개, 러닝 1개, 싸이클 1개 연동 MVP
- 수동 입력 폼 추가
- 중복 활동 방지용 `source_activity_id` 체계 도입

결과:

- 특정 커뮤니티에 묶이지 않는 구조 확보

## Phase 2. 클럽 플랫폼 기능 추가

목표:

- 여러 동호회가 각자 쓸 수 있는 운영 기능 확보

할 일:

- 사용자/클럽/멤버십 모델 도입
- 클럽별 홈 대시보드
- 모임 생성/참가 등록 기능
- 운영자 권한 체계 추가

결과:

- "내 프로젝트"가 아니라 "여러 클럽이 쓰는 서비스"로 전환

## Phase 3. 게임화 강화

목표:

- 재미 요소로 재방문율 강화

할 일:

- 스포츠별 배지 규칙 분리
- 주간 목표/월간 챌린지
- 레벨/칭호 체계
- 개인 역량 카드
- 클럽 시즌전/출석 이벤트

결과:

- 기록 플랫폼에서 커뮤니티 참여 플랫폼으로 확장

## Phase 4. 빌더화

목표:

- 운영자가 직접 플랫폼 구성을 바꿀 수 있게 함

할 일:

- 대시보드 위젯 ON/OFF
- 종목별 모듈 선택
- 배지 세트 선택
- 클럽 테마 설정
- 모임 운영 방식 선택

결과:

- "코드 수정 없이 동호회별 서비스 구성"이 가능해짐

## 10. 추천 우선순위

가장 중요한 순서는 아래다.

1. 입력 구조를 API/수동입력 기반으로 바꾸기
2. 데이터 모델을 수영 전용에서 멀티 스포츠 공통 모델로 바꾸기
3. 클럽/멤버십/권한 구조 추가
4. 모임 참가 기능 추가
5. 배지/목표 시스템을 종목 공통 규칙 엔진으로 확장
6. 마지막에 빌더형 관리 화면으로 고도화

이 순서가 중요한 이유는, 입력 구조와 데이터 모델이 바뀌지 않으면 이후 기능은 전부 수영 전용 구조에 묶이기 때문이다.

## 11. 제안하는 다음 작업

바로 다음 단계에서는 아래 3개를 먼저 진행하는 것이 좋다.

### A. 플랫폼 코어 설계 문서 작성

- 공통 activity 스키마
- sport_type별 확장 필드
- user / club / event 기본 ERD

### B. 현재 코드 구조 리팩터링안 작성

- 현재 모듈을 어떤 새 폴더 구조로 옮길지 정의
- 수영 전용 명칭 변경 범위 정리

### C. MVP 화면/기능 정의

- 클럽 홈
- 개인 프로필
- 모임 신청
- 챌린지/배지
- 운영자 설정

## 12. 최종 정리

현재 프로젝트는 이미 아래를 증명했다.

- 기록 수집이 가능하다.
- 기록을 집계하고 보여줄 수 있다.
- 랭킹과 개인 프로필을 만들 수 있다.
- 배지/칭호를 운영 규칙으로 계산할 수 있다.
- 관리자 설정으로 운영할 수 있다.

즉, 완전히 새로 만드는 것이 아니라
"현재 수영 대시보드를 플랫폼 코어로 승격시키는 작업"이 더 정확한 표현이다.

이 프로젝트의 다음 핵심 전환점은 아래 한 문장으로 정리된다.

"게시판 제목 파싱 기반 수영 대시보드"에서 "API 연동 기반 멀티 스포츠 동호회 플랫폼"으로 전환한다.
